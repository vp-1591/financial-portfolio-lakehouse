# 0066 — Portfolio Holdings Gold Table and Report Generation

> **Superseded by [ADR 0082](./0082-fold-portfolio-allocation-into-holdings.md)** — The `portfolio_allocation` fallback was removed; `portfolio_holdings` now includes percentages directly.

## Context

The reporting baseline roadmap (Phase 3) requires a self-contained HTML report showing
portfolio composition, passive income, cash flows, and data quality. Phase 2 introduced
CDC analytics gold tables (`dividend_income`, `interest_income`, `cash_flow_summary`)
that fully cover the CDC-based charts. However, the portfolio summary section needs
absolute `value` and `position_type` (EQUITY/CASH) — neither exists in any gold table.
`portfolio_allocation` has percentages only; `value` lives in encrypted form in
`consolidated_holdings` (normalized layer); `position_type` lives only in per-broker
`*_snapshot_normalized` tables. The roadmap mandates "gold layer is the single source of
truth" for the report, so reading normalized tables directly was not acceptable.

Additionally, no report generation code existed in the project — zero charting
libraries, HTML templates, or report subcommands.

## Decision

1. **Add a `portfolio_holdings` gold table** that joins `consolidated_holdings` (for
   base-currency value, already FX-converted in the consolidate step) with per-broker
   `*_snapshot_normalized` tables (for native-currency value, currency, and
   `position_type`). Join key is `(broker, ticker)` where `broker` matches
   `connector.display_name` and `ticker` matches the snapshot `label` column.
   Rows without a snapshot match get `position_type="UNKNOWN"` — they are never
   dropped.

2. **Reuse existing FX conversion** — `consolidated_holdings.value` is already in the
   target (base) currency because the consolidate step calls
   `CurrencyConverter.convert()`. The `portfolio_holdings` builder does not introduce
   any new currency conversion; it reads `value_base` from decrypted
   `consolidated_holdings.value` and `value`/`currency` from the snapshot.

3. **Build the report using Jinja2 + Plotly** — a self-contained HTML file with inline
   CSS and Plotly charts. The first chart inlines `plotly.js` (~3.5 MB); all subsequent
   charts use `include_plotlyjs=False` to avoid duplication. No server or CDN
   dependency.

4. **`pipeline report` subcommand** — reads gold tables via the existing DuckDB query
   connection (reuses S3 credentials), builds Plotly figures and HTML via Jinja2
   template, writes a local file. Configurable `--output` path, optional
   `--base-currency` display label, `--open` to launch browser.

5. **Report reads only gold tables** — the `portfolio_holdings` table (not raw or
   normalized tables) is the single source of truth for portfolio summary data. If
   `portfolio_holdings` is missing, the report falls back to `portfolio_allocation`
   (percentages only) and logs a warning. Missing CDC or data-quality tables produce
   empty sections, not errors.

## Constraints

- No new S3 credential mechanism — reuses `get_connection()` from `pipeline.query`.
- No server dependency — report is a static HTML file generated by a CLI command.
- No PDF or image export — HTML only.
- No market data — portfolio value over time is deferred to a future roadmap.
- Automated delivery (email, S3 upload, scheduled execution) is out of scope —
  deferred to productionization step 5.
- `position_type` is limited to EQUITY/CASH/UNKNOWN — finer classification (ETF,
  stock, bond, gold) requires market data metadata, deferred to the market data roadmap.

## Consequences

- **Positive**: The report is a single self-contained HTML file that opens in any
  browser with interactive charts. Gold-only data access keeps the medallion
  architecture clean. Adding `portfolio_holdings` makes the analytics layer complete
  for all current reporting needs.

- **Positive**: The `portfolio_holdings` table is reusable beyond the report — any
  future analytics or dashboard can query it directly.

- **Negative**: The HTML report is ~3.5–5 MB due to inlined Plotly.js. This is
  acceptable for local generation but may be large for email delivery (deferred to
  productionization).

- **Negative**: `position_type="UNKNOWN"` rows appear if a consolidated holding has no
  matching snapshot row. This should be rare (only if brokers diverge) but needs
  monitoring.

- **Follow-up**: When market data integration adds historical portfolio value, the
  report can extend to include a performance chart without changing the gold-layer
  access pattern.

## Validation

- `tests/test_portfolio_holdings.py`: 8 tests covering schema match, row count,
  decrypted values, position type (EQUITY/CASH), base currency, native value for
  EUR positions, Delta table write, and missing-table error handling.

- `tests/test_report.py`: 6 integration tests covering HTML generation, section
  markers, Plotly chart embedding, partial data rendering, empty-data error, and
  default output path.

- `pipeline report --output data/report.html` on demo data produces a visible HTML
  report with all four sections rendering interactive charts.

- All 536 existing tests pass with no regressions.
